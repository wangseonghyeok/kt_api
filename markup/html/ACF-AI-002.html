token 기본규격

API 이름
접근토큰 요청(token)
API URL
~/kcert/v1.0/token
METHOD
POST
API 설명
  • 표준 API 사용 승인을 위한 접근토큰을 발급 받는다.
  • 발급된 접근토큰은 향후 API 접근 시 Authorization header에 삽입하여 API 사용 승인을 수행한다. (단, clientrequest 제외)
  • 접근토큰은 유효기간 만료까지 여러 트랜잭션에서 사용 가능하다.
  • (예시) Authorization : Bearer {access_token}

prepare 기본규격

API 이름
사전 준비 요청(prepare)
API URL
~/kcert/v1.0/prepare
METHOD
POST
API 설명
  • 간편인증 트랜잭션의 최초 요청 단계이다.
  • 간편인증을 요청하기 위해 다음의 2가지 방법을 사용할 수 있다.
    1) request API 사용
    2) prepare API + clientrequest API 사용
    - 인증앱에서 사용자가 전자서명 원문을 확인하기 어려운 경우
    - 사용자는 이용기관 앱에서 전자서명 원문을 확인하고 직접 전자서명인증사업자의 서버로 서명 원문의 해시값을 전달(clientrequest)함으로써 전자서명인증사업자의 서버에서 전자서명 원문의 위변조를 확인할 수 있다.
    ※ 이용기관이 보내온 전자서명 원문을 해시하여 사용자가 직접 보내온 전자서명 원문의 해시값과 비교
  • 전자서명인증사업자는 clientrequest에서 사용할 일회용 접근토큰을 발급한다.
  • clientrequest 전에 반드시 호출되어야 한다.
  • 수신된 일회용 접근토큰은 clientrequest 호출 시에 1번만 사용 가능하다.
  • 이용기관은 사용자 정보, 전자서명 원문, 서명방법 등의 정보로 간편인증을 요청하고, 전자서명인증사업자는 TransactionID(tx_id) 값과 인증앱 호출 방법 등을 응답한다.
  • 전자서명인증사업자는 prepare 요청에 푸시 알림 요청이 있더라도 알림을 바로 발송하지 않고, clientrequest 요청을 받은 후에 푸시 알림을 발송해야 한다.

clientrequest 기본규격

API 이름
클라이언트 서명 요청(clientrequest)
API URL
~/kcert/v1.0/clientrequest
METHOD
POST
API 설명
  • 간편인증 트랜잭션의 최초 요청 단계로 prepare의 후속 처리 과정이다.
  • 간편인증을 요청하기 위해 다음의 2가지 방법을 사용할 수 있다.
    1) request API 사용
    2) prepare API + clientrequest API 사용
    - 인증앱에서 사용자가 전자서명 원문을 확인하기 어려운 경우
    - 사용자는 이용기관 앱에서 전자서명 원문을 확인하고 직접 전자서명인증사업자의 서버로 서명 원문의 해시값을 전달(clientrequest)함으로써 전자서명인증사업자의 서버에서 전자서명 원문의 위변조를 확인할 수 있다.
    ※ 이용기관이 보내온 전자서명 원문을 해시하여 사용자가 직접 보내온 전자서명 원문의 해시값과 비교
  • prepare 단계에서 발급받은 일회용 접근토큰을 통해 API 인증을 수행한다.
    Authorization: Bearer {one_time_token}
  • clientrequest 전에 prepare 호출이 선행되어야 한다.
  • 전자서명인증사업자는 일회용 접근토큰을 검증하여 API 접근권한을 확인한다.
  • 사용자 화면(웹/앱)에서 확인된 최종 원문의 해시값을 전자서명인증사업자의 서버로 전송하고, 전자서명인증사업자는 prepare 단계에서 이용기관으로부터 받은 원문과 clientrequest 단계에서 받은 해시값을 검증함으로써 원문의 위변조 여부를 확인한다.
  • 전자서명인증사업자는 prepare 요청에 PUSH 알림 요청이 있는 경우, clientrequest 수신 후에 PUSH 알림을 발송한다.

request 기본규격

API 이름
서명 요청(request)
API URL
~/kcert/v1.0/request
METHOD
POST
API 설명
  • 간편인증 트랜잭션의 최초 요청 단계이다.
  • 간편인증을 요청하기 위해 다음의 2가지 방법을 사용할 수 있다.
    1) request API 사용
    - 간편인증(단순 로그인)의 경우
    - 전자서명을 수행하는 인증앱에서 원문을 확인하는 경우
    2) prepare API + clientrequest API 사용
  • 이용기관은 전자서명을 요청하는 사용자 정보, 전자서명 원문, 서명방법 등의 정보로 전자서명인증사업자에게 간편인증을 요청하고, 전자서명인증사업자는 TransactionID(tx_id) 값과 인증앱 호출 scheme 등을 이용기관에 응답한다.
  • 전자서명인증사업자는 request 요청에 푸시 알림 요청이 있는 경우 요청을 받은 후에 푸시 알림을 즉시 발송한다.

result 기본규격

API 이름
결과 요청(result)
API URL
~/kcert/v1.0/result
METHOD
POST
API 설명
  • 사용자의 전자서명 결과값 요청
  • 사용자의 전자서명이 완료된 경우, 서명값을 수신
  • 서명이 완료되지 않은 경우, 진행 상태값과 각 이전 상태의 처리 시간을 수신

verify 기본규격

API 이름
검증 요청(verify)
API URL
~/kcert/v1.0/verify
METHOD
POST
API 설명
  • 전자서명을 검증하기 위한 용도로 사용
  • 이용기관에서 전자서명 원문을 자체 검증하지 않는 경우, 전자서명인증사업자에게 result로 전달 받은 전자서명 원문을 전달하여 검증 요청
  • 전자서명인증사업자는 전자서명 검증 결과와 인증서 유효성 검증 결과를 응답

validate 기본규격

API 이름
인증서 유효성 검증 요청(validate)
API URL
~/kcert/v1.0/validate
METHOD
POST
API 설명
  • 인증서의 유효성을 확인하는 용도로 사용
  • 이용기관에서 전자서명원문을 자체 검증 후, 인증서의 유효성을 확인하기 위해 사용

identify 기본규격

API 이름
서명자 확인 요청(identify)
API URL
~/kcert/v1.0/identify
METHOD
POST
API 설명
  • 서명자의 신원을 확인하기 위한 용도로 사용
  • 전자서명인증사업자는 전자서명 검증 결과, 인증서 유효성 검증 결과, 서명자 확인 정보를 응답
  • VCI는 이용기관이 기 보유중인 연계정보와 비교/검증하는 용도로 사용
